Scenario

Our admin team would like to collect information about our client's subscription habits for business decision making.

Task

Share with us how would you implement the scenario - (you are free be as creative as you wish but robustness, practicality and simplicity would suffice)

Questions of interests:

  • What would you implement? (Assuming a database and CRM of all the clients already exists)
  • How would you implement? (Tech-stacks used, any special features, computation algorithms etc.)
  • What kind of value would the admin team be able to get? (Hint: Tracking information is always useful)

Answer

  • We can already gather a lot of information by utilizing existing tools such as Google Analytics. This provides a variety of data including:
    • Traffic sources
    • Page hits
    • Path exploration
    • Browser information
    • Country of origin
    These insights can be used in conjunction with our client database to better profile and map our target markets. Additionally, analyzing:
    • Page hits
    • Path explorations
    allows us to monitor which features are being used most frequently or are enticing clients to subscribe. This information helps us understand what to prioritize for prospective clients and provides suggestions for improvements or upgrades.

  • From our existing database, we can generate reports to monitor subscription habits. This involves adding attributes such as:
    • Date of subscription/cancellation
    • Renewal counts
    • Whether a discount/promotion/bundle was considered
    To achieve this, we ensure our database is properly time-stamped and tracks as many attributes as possible. Such functionality is available in various frameworks like Laravel or Django, either natively or through plugins.

  • Adding audits to our platform enhances our ability to monitor user activity in greater detail. Audits allow us to track user actions as they navigate the system, including:
    • Updating their information
    • Login times
    • Pages/modules accessed
    This functionality can be incorporated using plugins available in frameworks such as Django or Laravel.

Answer

  • Ideally the we have a separate centralized database server for the platforms to connect to.
  • While each platfomr might have different ways of storing clients data as in different attributes or differing values depending on the platform the client data can still using a Master Table that is reffrenced by each platform.
  • This will allow each platform to be customizable while still have only a single master table for the client information to ensure data consistency.
  • We can clean the data by identifying the the essential values of the client(ID, names or any universally used values) and use that as the master table which then the separate platform can assign attributes and values specific to them
  • This way we can avoid data duplication and make sure data consistency between platform and simplifies maintenance and updates by only having 1 source.

Scenario

A free finance news API allows 10 calls per minute. Each API call would take in a financial keyword (ie. inflation) and would take 5 seconds to process its sentiment analysis to be sent to the end-user. As the number of end-users increases, the frequency of calls increases greatly.

Task

Suggest one or more implementations that would allow the end-user to still receive the requested info with lesser disturbances or delay.

Answer

  • Best method is using caching to increase responsiveness and reduce the resources used by the system by having the data stored somewhere and can be retrieved immediately instead of doing the process over and over again.
  • Some frameworks like Django have server side caching which you can configure to use whatever method is most suitable (memory, file, database(regular or redis)) depending on the scale and usage of the application.
  • In this case when a user or another user try to request another keyword that have been requested before, instead of taking 5 seconds to do the process, it can just send the data to the uses.
  • One thing to keep in mind is on the source data volatilty. Caching might not the best solution to highly volatile data since it will cause the data to be stale. One way to circumvent this(depending on the nature of the api) is to have a list of popular keyword and have trigger to prefecth the data for them when something in the backend is updated.
